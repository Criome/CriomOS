# CriomOS Deep Audit — 2026-04-17

Full codebase audit covering flake inputs, NixOS modules, home-manager,
zone/sphere builders, network, security, deployment, and packaging.

Findings are grouped by severity, then by subsystem.

---

## Critical

### SEC-1: LLM API key visible in process list
`nix/mkCriomOS/llm.nix:132` — `--api-key ${apiKey}` passed on the command
line. Any user can see it via `ps aux`.

**Fix:** Use `EnvironmentFile=` or systemd's `LoadCredential=` to inject the
key from a file, not the command line.

### SEC-2: WiFi SAE password hardcoded in version control
`nix/mkCriomOS/router/default.nix:116` — `saePasswords = [{ password = "leavesarealsoalive"; }]`
is committed to the repo and baked into `/nix/store/`.

**Question:** Is this intentional scaffolding or should it move to
sops/agenix? Even for a private LAN, the password is in GitHub history.

### SEC-3: Private keys passed via shell variables
`nix/mkCriomOS/network/nordvpn.nix:44,60` and `network/wireguard.nix:47` —
WireGuard/NordVPN private keys read into shell variables and passed to
processes. Visible in `/proc/<pid>/environ` during generation.

**Fix:** Write keys to tmpfiles with 0600 permissions and pass file paths, or
use systemd `LoadCredential=`.

## High

### FLAKE-1: Nixpkgs version divergence across inputs
Four distinct nixpkgs revisions coexist in `flake.lock`:
- Root: `b40629e` (2026-03-18)
- claude-code's nixpkgs: `8d8c1fa` (2026-04-02) — newer than root
- crane (via mentci): `9b008d60` (2025-07-08) — 9 months stale
- attic's crane: own nixpkgs, not following root

**Question:** Is the stale crane nixpkgs causing eval divergence? Should
`attic.inputs.nixpkgs.follows = "nixpkgs"` be added?

### FLAKE-2: Quadruplicate flake-utils and triplicate crane
Five `flake-utils` instances and three `crane` instances in `flake.lock`, all
resolving to similar or identical revisions. Missing `follows` declarations
cause unnecessary duplication during eval.

**Fix:** Add `follows` for flake-utils and crane in transitive inputs that
don't already have them.

### MOD-1: metal/default.nix is 503 lines of mixed concerns
Firmware, kernel modules, power management, udev rules, ACPI handlers,
battery control, model-specific boot params — all in one file. Model-specific
logic is scattered across 8+ conditional blocks.

**Suggestion:** Split into:
- `metal/firmware.nix` (firmware index, enableAllFirmware)
- `metal/power.nix` (powertop, battery-ctl, EPP, lid switch)
- `metal/thinkpad.nix` (T14 Gen2, E15 Gen2, X240/X250 specifics)
- `metal/gpu.nix` (amdgpu params, TTM, Vulkan)

### MOD-2: enableAllFirmware contradicts modelFirmwareIndex
`metal/default.nix:77-83` defines per-model firmware lists, but line 201 sets
`hardware.enableAllFirmware = true` with a TODO comment. The model-specific
lists are dead code.

**Question:** Should enableAllFirmware be false, using only the model lists?
Or remove the model lists?

### MOD-3: Duplicate module selection logic
`nix/mkCriomOS/default.nix:58-65` and `:82-89` — identical conditional
module lists for `nixosModules` and `vmModules`. DRY violation.

**Fix:** Extract shared module list into a `let` binding.

### SEC-4: SSH hardening incomplete
`normalize.nix:142-147` enables OpenSSH with `PasswordAuthentication = false`
but leaves defaults for:
- `PermitRootLogin` (defaults to `prohibit-password` — allows key-based root)
- `X11Forwarding` (defaults to true)
- `AllowTcpForwarding` (defaults to true)
- `AllowAgentForwarding` (defaults to true)
- `ClientAliveInterval` (not set)

**Question:** Is key-based root login intentional (needed for deploy.sh)?
Should X11 forwarding and TCP forwarding be disabled?

### SEC-5: No kernel/network hardening sysctls
No `net.ipv4.tcp_syncookies`, `net.ipv4.conf.all.rp_filter`,
`kernel.dmesg_restrict`, `kernel.kptr_restrict`,
`kernel.unprivileged_userns_clone` restriction, or ICMP rate limiting.

**Question:** Is this acceptable given the private Yggdrasil mesh, or should
baseline hardening be applied regardless?

### DEPLOY-1: reload-shell.sh race condition
`src/criomos-deploy/reload-shell.sh:38-40` — kills quickshell, sleeps 1s,
launches noctalia-shell. No guarantee the process exited. No verification
that the new shell started.

**Fix:** Wait-loop on `kill -0` before restart. Check `pgrep` after launch.

### HM-1: Inverted dunst enable logic
`nix/homeModule/min/default.nix:438` — `dunst.enable = !sizedAtLeast.min`
disables dunst on min-or-larger nodes (the opposite of what's intended).
Settings are defined but never used.

**Question:** Should this be `enable = sizedAtLeast.min`? Or is dunst
replaced by noctalia's notification system on edge nodes?

### SCHEMA-1: behavesAs incomplete — 2 species missing
`nix/mkCrioZones/mkHorizonModule.nix:209-219` — `mediaBroadcast` and
`routerTesting` appear in the species enum (`speciesModule.nix:37-47`) but
are absent from all `behavesAs` derivations. A node with either species would
get no behavior flags.

**Fix:** Add rules:
```nix
mediaBroadcast = typeIs.mediaBroadcast;
routerTesting = typeIs.routerTesting;
```
And wire them into `router` and/or `edge` as appropriate.

## Medium

### FLAKE-3: 14+ flake inputs with `flake = false` may be unused
Emacs source inputs (gptel, jujutsu-el, md-roam, superchat, tera-mode,
ultra-scroll, xah-fly-keys, etc.) are declared in `flake.nix` and passed
through `hob`, but some may not be consumed by any module.

**Question:** Are all of these still used in mkEmacs or homeModule? A grep
for each input name in Nix code would confirm.

### MOD-4: normalize.nix is a catch-all (176 lines)
Boot tuning, SSH, PipeWire, NetworkManager, udev, and user defaults all live
in one file. Hard to reason about scope.

**Suggestion:** Split into `boot.nix`, `security.nix`, `network/base.nix`,
`desktop.nix`.

### MOD-5: horizon.json exported in two places
Both `normalize.nix:82-85` and `nix.nix:107-110` write the same
`horizon.json` file with the same mode. Redundant.

**Fix:** Keep one, remove the other.

### MOD-6: Llama service lacks hardening
`nix/mkCriomOS/llm.nix:119-146` — the llama-server service has
`MemoryMax=110G` (good) but no `NoNewPrivileges`, `ProtectHome`,
`ProtectSystem`, `SystemCallFilter`, or `PrivateTmp`.

**Suggestion:** Use yggdrasil's service hardening
(`network/yggdrasil.nix:93-109`) as a template.

### MOD-7: Headscale service missing Restart policy
`network/headscale.nix:108-111` — no `Restart =` directive. If headscale
crashes, it stays down.

**Fix:** Add `Restart = "on-failure"; RestartSec = 5;`.

### MOD-8: hasTouchpad defined but never used
`metal/default.nix:73` — `hasTouchpad = true;` with a TODO. Not referenced
anywhere.

**Fix:** Remove or wire into touchpad configuration.

### DEPLOY-2: deploy.sh temp file collision
`src/criomos-deploy/deploy.sh:54` — `/tmp/criomos-deploy` used as a fixed
path. Parallel deploys to the same node race on this file.

**Fix:** Use `mktemp -d` and an exit trap for cleanup.

### DEPLOY-3: No pre-deploy validation
Neither `criomos-deploy` nor the manual workflow runs `nix flake check`
before building. Evaluation errors are caught only during `nix build`.

**Question:** Should `criomos-deploy` do a local `nix eval` of the target
attr before SSH-ing to the node?

### HM-2: Hardcoded lat/lng in darkman config
`nix/homeModule/baseModule.nix:251-252` — `lat = 36.7; lng = -4.4;` fixed
coordinates. Geoclue is enabled and works, so these are fallback-only, but
they're wrong for any node not in southern Spain.

**Fix:** Remove hardcoded values or derive from node/cluster data. Geoclue
should be the sole source.

### HM-3: Hardcoded /home/ prefix
`nix/homeModule/baseModule.nix:222` — `homeDirectory = "/home/" + user.name`.
Breaks if a user has a non-standard home.

**Question:** Is this ever a problem in practice? NixOS always uses /home/.

### HM-4: Wallpaper resolution hardcoded
`nix/homeModule/baseModule.nix:284-288` — generates a 1920x1080 solid-color
PNG. Doesn't adapt to display resolution.

**Question:** Does noctalia/niri scale this, or does it display at native
resolution?

### HM-5: med/element.nix defines a system service in home-manager
`nix/homeModule/med/element.nix:8-52` — defines `systemd.services.nginx-element`
in a home-manager module. This is the wrong layer. Also references
`pkgs.nginx...element` which doesn't exist.

**Question:** Is this dead code? Should it be removed or moved to mkCriomOS?

### SCHEMA-2: hasVideoOutput hardcoded to behavesAs.edge
`mkHorizonModule.nix:221` — `hasVideoOutput = behavesAs.edge`. A center node
with a display (workstation) would be treated as headless.

**Question:** Should this be a node-level field in datom.nix instead of
derived from species?

### SCHEMA-3: io.bootloader and io.swapDevices defined but never consumed
`speciesModule.nix:147-167` defines `bootloader` and `swapDevices` in
IoOptions, but no module reads them. Disk/swap config comes from
NixOS hardware-configuration, not horizon.

**Question:** Are these intended for future use, or dead schema?

### SCHEMA-4: domains field defined but never wired
`clustersModule.nix` defines `domains` (attrsOf domainSubmodule) with
`species: "cloudflare"`, but no module consumes it. Expected use: DNS zone
management.

**Question:** Planned feature or dead schema?

### SCHEMA-5: sizedAtLeast.max uses `== 3` instead of `>= 3`
`criomos-lib.nix:69` — `max = size == 3`. If the magnitude enum ever
grows beyond 3, this breaks. Currently safe since enum is [0,1,2,3].

### PKG-1: 6 unused packages in adHoc.nix
`nix/pkdjz/adHoc.nix` defines packages that are never referenced:
- `base16-styles` (line 11-26)
- `jumpdrive` (line 32-101)
- `davinci-resolve` (line 254-293)
- `skylendar` (line 296-310)
- `videomass` (line 400-432)
- `wireguardNetresolved` (line 434-457)

~250 lines of dead code.

### PKG-2: Broken reference to pkdjz.crate2nix
`nix/homeModule/neovim/default.nix:301` references `pkdjz.crate2nix` which
is not defined anywhere in pkdjz.

### PKG-3: Insecure olm-3.2.16 permitted
`nix/mkPkgs/default.nix:13` — `permittedInsecurePackages = ["olm-3.2.16"]`
with a TODO comment.

**Question:** Is olm still needed? Can it be updated or removed?

## Architecture Questions

### Q1: Should CriomOS adopt sops-nix or agenix?
Three secret categories exist (WiFi passwords, WireGuard keys, NordVPN keys)
with no unified management. All use ad-hoc file placement with varying
permission models. A secrets framework would resolve SEC-1 through SEC-3.

### Q2: Should pkdjz be replaced by direct flake input consumption?
Memory note says "pkdjz indirection layer is discouraged — use flake inputs
directly." Yet pkdjz is actively used (mkEmacs, evalNixos, kynvyrt, etc.).
Is the plan to migrate these to direct inputs, or is pkdjz staying?

### Q3: Should normalize.nix be decomposed?
It's 176 lines of mixed concerns. Decomposing would improve readability but
add more files. Is the current size acceptable?

### Q4: Should metal/default.nix get a model dispatch table?
Instead of scattered `mkIf (model == "ThinkPadT14Gen2Intel")` blocks,
a model-keyed attrset could centralize per-model config:
```nix
modelConfig.ThinkPadT14Gen2Intel = { kernelModules = ...; bootParams = ...; };
```
This would make adding new models mechanical.

### Q5: Should horizon export an explicit hasDisplay field?
Currently `hasVideoOutput = behavesAs.edge` which is a heuristic. Adding a
boolean to datom.nix would be more accurate and support edge cases
(headless edge, center with display).

### Q6: What's the plan for distributed builds?
`nix.nix:169-171` marks them "broken". Is this a priority? The
infrastructure (nix.sshServe, nixdev group) is partially wired.

### Q7: Should the domain schema in clustersModule be wired or removed?
It defines Cloudflare domain types but nothing consumes them. Either wire
to DNS/cert management or remove to avoid schema drift.
